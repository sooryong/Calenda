# 일정 분류 기준 (단일 SOT — generator·evaluator·golden 공용)

> `schedule_status` 분류의 **유일한 기준**. generator(생성)·evaluator(QA)·golden(라벨)이 모두 따른다.
> **3-분류**: `schedule_status: "yes" | "pending" | "no"`. A(단일 메시지)·B(멀티턴 대화)는 다른 잣대를 쓴다.

---

## 0. 3-분류 정의 + 공통 원칙

| 값 | 의미 | events | 앱 |
|---|---|---|---|
| **yes** | 확정 일정 — 사용자가 **당사자로 확실히 참석/수행**, 날짜·시각 분명 | 추출 | 자동등록 |
| **pending** | **미래 행사/행동 가능성** — 공고·모집·마감·설명회·포럼·초대·미확정 제안 (확정 아님, 사용자 판단 필요) | 추출 | 예비 |
| **no** | 비-일정 — 거래/통보/광고/인사/과거/남의 일정 | [] | 무시 |

★ **핵심 판별: "사용자에게 미래 행사·행동이 있나? 그게 확정인가?"**
- 확정(내가 확실히 감/함) → **yes**
- 가능성 있음(갈 수도·신청할 수도 — 미확정) → **pending**
- 없음(자동 거래·통보·광고) → **no**

★ **날짜·시각의 존재는 분류 근거가 아니다.** "적립일시 6/15 10:43"·"결제예정일 6/17"은 날짜·시각이 있어도 **no**(거래통보). "끝나고 보자"(시각 없음)도 약속이면 yes/pending.
★ 순서: **① yes/pending/no 분류 → ② yes·pending이면 date/time 정확히 추출.** (confidence 없음 — status가 확실성·라우팅을 표현.)
★ 각 판정은 `_label_reason`(어느 기준 항목인지 한 줄)을 남긴다.

## A. 단일 메시지 — "내용의 종류"로 분류

### A-yes: 확정 일정
사용자가 당사자로 **확실히 참석/수행**할, 일시가 정해진 일정:
- **개인 약속**: "내일 3시 강남역에서 봐요", "내일 12시 외할머니댁, 저녁 7시 이모네 모임"
- **기관 예약·확정 알림**: 의료("[병원] 6/3 10:30 진료 예약"), 항공·철도("KE093 6/14 18:30 출발 확정"), 숙박·식당 예약, 면접("1차 면접 6/16 14:00 ○○빌딩"), 방문 서비스("기사 6/17 오전 방문 예정")
- 판별: **이미 정해졌고, 내가 그 시각에 갈/할 일이다.**

### A-pending: 미래 행사/행동 가능성 (확정 아님 → 사용자 판단)
사용자가 관심 있으면 참석·신청할 수 있으나 **참석/실행이 확정은 아닌** 것 → 예비로 위임:
- **행사 안내·참석 요청**: 설명회·포럼·세미나·간담회·출범식·공유회·총회·교육 ("전문가 대상 설명회 1/28(수) 14시", "인스트럭터 포럼 11/2 16:00", "성과공유회 12/9 참석 요청", "출범식 6/16 14:00")
- **모집/공고/마감/신청/납부**: "국가장학금 신청 마감 6/19", "교육 모집 ~11/30", "등록금 납부 7일", "건강검진 12/31까지", "도서 반납 6/15"
- **조인/참가 요청**: "골프조인 요청서 6/27 05:40", 선착순 행사 신청 권유
- 판별: **미래에 갈/할 수 있는 행사·행동이 있으나, 내가 할지는 미정** → 예비.
- ★ **pending은 구체 미래 날짜(마감일·납부일·행사일·신청일)가 있어야 한다.** 날짜가 전혀 없는 공고·안내·모집은 등록할 이벤트가 없으므로 **no**. (개인 신청·신고·납부·마감·등록도 날짜 있으면 pending. 단 구체 *시각*까지 정해진 개인 행동·약속은 yes — 예: "내일 15시 헬스장 등록", "12/19 송년회".)

### A-no: 비-일정 통보
미래 행사·행동이 없는 것(자동 거래·통보·정보):
- **거래/금융**: 카드 승인·출금·입금·결제예정·정기결제·청구서
- **적립/포인트/멤버십/구독/쿠폰**, **배송/물류**(도착 통보), **인증/보안**(로그인·인증유효시각)
- **영업/상담 로지스틱스**: "오후에 연락드릴게요", "견적/사진 보내드릴게요", "검토 후 회신"
- **광고/홍보**, **인사말·안부·잡담**, **단순 정보·뉴스·시행일**("6/17부터 정책 변경")
- **과거 이벤트 회고**("어제 14시 세미나 잘 마쳤습니다"), **남의 일정 언급**("내일 종일 간담회라 정신없을 듯")

### A-경계
- ★ **거래 "안내"라도 미래 행사/행동이 없으면 no.** "적립 안내"·"결제 안내"·"배송 안내" = no. "행사/설명회 안내"·"모집 안내"는 **구체 날짜가 있으면 pending, 날짜 없으면 no**.
- ★ **개인 마감/할일(신고·납부·신청·등록·반납·검진)**: 날짜 있으면 **pending**(예: "10일 종합소득세 신고", "9일 자동차세 납부", "25일 여권 재발급 신청"). 단 **거래 완료 통보**("자동납부 완료"·"자동이체 등록 완료")·광고·취소·과거는 **no**.
- 배송 **"방문 수령"** + 일시 명시(사용자가 가야 함) → yes / 단순 도착 통보 → no.

## B. 멀티턴(대화) — "확정 도달 여부"로 분류

확정은 메시지 내용이 아니라 **대화 상태**에서 나온다.

### B-yes: 확정
- 양쪽이 특정 날짜·시각에 **합의**, 마지막이 수락/재확인 ("좋습니다 / 네 화요일 13시에 봬요 / 내일 뵙겠습니다 / Confirmed").
- ★ **하나의 약속 = 한 건. 최종 합의값** 추출(중간 제안 말고 마지막 합의). 재확인/링크공유도 그 약속을 가리킴.

### B-pending: 미확정 제안
- 구체 시각 **제안으로 끝남** ("화요일 13시 어때요?", "6/4 9시 가능하세요?") — 아직 수락 안 됨. 사용자가 결정할 여지 → 예비. (제안된 날짜·시각 추출)

### B-no: 무산
- **거절/취소** ("다른 일정 있어 다음에", "회의 취소합니다"), **유보/모호**("조만간 보자" — 구체 시각 없음), 협의 무관 잡담.

## 대조 예시 (같은 표면형, 다른 라벨 — 토큰 의존 차단)

| 메시지 | 라벨 | 이유 |
|---|---|---|
| "내일 14시 강남역에서 봐요" | **yes** | 확정 약속, 내가 감 |
| "[병원] 06.03 10:30 진료 예약 안내" | **yes** | 확정 예약 |
| "전문가 설명회 1/28(수) 14시 안내" | **pending** | 행사, 참석 미확정 |
| "국가장학금 신청 마감 6/19" | **pending** | 행동 가능, 미확정 |
| "골프조인 요청 6/27 05:40" | **pending** | 참가 요청, 미수락 |
| "카카오페이 적립 6/15 10:43" | **no** | 거래 통보 |
| "Buddy Pass 결제예정일 6/17" | **no** | 자동 결제 |
| "오후에 연락드릴게요" | **no** | 로지스틱스 |
| "…화요일 13시 봬요"(확정) | **yes** | 대화 확정, 최종 합의 |
| "화요일 13시 어때요?"(제안) | **pending** | 미확정 제안 |
| "다른 일정 있어 다음에" | **no** | 거절 |

## 출력/QA 규약
- 각 판정에 `_label_reason` 한 줄(어느 기준 항목인지).
- **이중 합의**: generator가 라벨 → evaluator가 독립 재판정, 불일치는 reject/fix(기준 일관성 강제).
- **confidence 없음** — status(yes/pending/no)가 확실성·라우팅을 표현. (옛 "날짜+시간 → 0.9" 신뢰도 루브릭 폐기 — date/time 지름길 편향의 원천이었음.)

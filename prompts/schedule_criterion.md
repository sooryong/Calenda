# 일정 분류 기준 (SOT — generator·evaluator·golden·audit 공용)

`schedule_status` ∈ `"yes" | "pending" | "no"`. 아래 **두 질문으로** 분류한다.

## 존재 이유 (설계 원칙)

> **"현재 시각 이후, 사용자 본인이 해야 할 일·지켜야 할 약속의 제목·날짜·시간·장소를 찾는다."**

| 값 | 의미 | events | 앱 |
|---|---|---|---|
| **yes** | 내가 할 일·약속·참석/업무 안내 (현재 시각 이후) | 추출 | 자동등록 |
| **pending** | 불특정 다수 대상 공고·안내 / 내게 오는 배송 예정 (관심은 있으나 직접 수행하는 약속은 아닌 것) | 추출 | 예비 — 사용자가 yes 선택 가능 |
| **no** | 비-일정 | `[]` | 무시 |

---

## 분류 테스트

**Q0. 수신 시각 이후의 일인가?**
- **아니오 → `no`** — 수신 시각 기준 이미 지나간 이벤트. "어제 회의 잘 됐어요", "결제가 완료되었어요", 오늘 이미 지난 시각의 이벤트 등. **과거형 동사("~됐어요", "~완료되었어요", "~마쳤습니다") 포함.**

**Q1. (Q0이 '예') 어떤 종류인가?**

- **→ `yes`**: 나를 특정해서 요청·안내·합의된 것
  - 내가 직접 참석·수행하는 약속·회의·면접·진료예약
  - 업무 요청·참석 안내 ("6/26 강의 부탁드립니다", "회의 참석 바랍니다")
  - 멀티턴 대화에서 합의에 도달한 약속
  - "뵙겠습니다", "하겠습니다" 단정형
  - ★ **yes 필수 조건**: 제목(무엇을) + 날짜·시간 + 장소 중 최소 제목+날짜·시간이 명확해야 자동등록 가능. 하나라도 불명확하면 pending.

- **→ `pending`**: 직접 수행하는 약속은 아니지만 알고 있어야 하거나 수락 여부를 결정해야 하는 것
  - 설명회·행사·교육 개최 안내 (불특정 다수 발송) — 나의 일정은 아니지만 인지할 필요
  - 모집·지원사업·공고 (신청 가능) — 나의 일정은 아니지만 인지할 필요
  - **배송 예정 알림** (날짜·시간이 있고 **수령인 이름이 사용자와 일치**하는 수령 대기)
  - **미수락 제안·단체공지** ("금요일 7시 동기회", "만나자", "ㄱㄱ" 등 — 사용자가 수락 여부 미확정)

- **→ `no`**: 나머지 전부
  - 거래·결제·적립·인증
  - 배송 완료 알림 (과거형) / 주문 확인 (배송 일정 미확정)
  - 광고·홍보 (서비스 프로모션, 할인 안내 등)
  - 인사·안부·영업 연락
  - 남의 일정 언급 ("저 오늘 회의 있어서...") — **발신자 본인이 자신의 일정을 맥락으로 언급** ("마치고 연락드릴게요", "끝나고 확인할게요" 등)
  - 단체방에서 **@타인**에게 하는 업무 지시·강의 요청 — 사용자(수신자)가 아닌 제3자에 대한 요청
  - 단순 정보·정책 안내 (내가 할 행동 없음)
  - 거절·취소·완료 알림
  - 예약확인이지만 수신자(탑승객·투숙객) 이름이 사용자와 불일치
  - 예약확인에 탑승객·투숙객·예약자 이름이 아예 없는 경우 (본인 확인 불가)
  - 배송 예정이지만 수령인 이름이 없거나 사용자와 다른 경우
  - **선택적 참석 안내** — "참석이 필수 사항은 아니나", "참석은 선택", "참석을 환영하나 필수는 아닙니다" 등 참석 의무가 없는 행사 안내. 회신 요청이 포함되어 있어도 no.
  - **날짜 미확정 언급** — "아마 다음 주부터", "곧 시작", "추후 안내", "미확정" 등 구체적 날짜·시각이 없는 막연한 일정 언급 → no.
  - **일정 확정 없는 회신 요청** — "편하신 시간에 회신주시면", "가능하신 날짜 알려주시면" 등 날짜·시각이 확정되지 않고 사용자에게 가용 시간을 묻는 것 → no. (날짜·시각이 명시된 강의/미팅 수락 요청은 pending)
  - **구글캘린더 알림** — 발신자가 `Google Calendar` 또는 `calendar-notification@google.com`이거나, 본문에 "캘린더 알림을 구독하셨기 때문에" 포함 → 이미 캘린더에 등록된 일정의 리마인더이므로 재등록 불필요. 본문에 장소·참석자·설명 등 상세 정보가 풍부해도 no.

★ **멀티턴**: "확정"은 대화가 합의에 도달했는지로 본다 — 제안만 하고 끝 = pending, 거절·취소·모호("조만간 보자") = no.
★ **타인 일정 언급**: 발신자가 자신의 일정을 맥락으로 말하는 것("저 오늘 회의 있어서 나중에 연락드릴게요")은 Q1=no. 사용자 일정이 아님.
★ **자기발신 SMS**: 발신자 번호 = 사용자 본인 번호이면 자기 메모 → Q1=yes. 모델 입력에 `<사용자 전화번호>` 주입 필요.
★ **이름·번호 불일치**: 예약확인 메시지의 탑승객/투숙객/예약자 이름이 사용자 이름(또는 전화번호)과 다르면 타인 예약 → Q1=no. 모델 입력에 `<사용자>` 이름 주입 필요.
★ **날짜·시각 표면 토큰은 판단에 쓰지 않는다.** date는 yes·pending일 때만 추출(없으면 null). no면 `events: []`.

## 앱-모델 역할 분담

모델은 Q0~Q1 기준으로 분류·추출. 최종 등록 결정은 앱이 보정:

| 모델 출력 | 앱 처리 |
|---|---|
| yes | 자동등록 |
| yes (현재 이전, 오류) | pending 강등 (EventRouter 안전망) |
| pending | 예비 알림 — 사용자가 yes 선택 가능 |
| no | 무시 |

## QA
각 판정에 `_label_reason` 한 줄(Q0/Q1 어디서 갈렸는지). generator 라벨 → evaluator 독립 재판정, 불일치 reject/fix.

> 단서: "강의하셔야 하는 상황입니다"처럼 Q1 경계가 애매한 케이스는 남는다(0.6B 천장). 나를 특정한 요청이면 yes, 다수 대상이면 pending으로 판단.

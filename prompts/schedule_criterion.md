# 일정 분류 기준 (SOT — generator·evaluator·golden·audit 공용)

`schedule_status` ∈ `"yes" | "pending" | "no"`. 아래 **세 질문으로** 분류한다.

## 존재 이유 (설계 원칙)

> **"수신 시각 이후, 사용자 본인이 해야 할 일·지켜야 할 약속의 제목·날짜·시간·장소를 찾는다."**

| 값 | 의미 | events | 앱 |
|---|---|---|---|
| **yes** | 확정 일정 (내일 이후·오늘 미래 시각, 확정) | 추출 | 자동등록 |
| **pending** | 미확정 가능성 (오늘 미래 시각 확정·안내·제안·신청) | 추출 | 예비 |
| **no** | 비-일정 | `[]` | 무시 |

---

## 분류 테스트

**Q0. 수신 시각 이후의 일인가?**
- **아니오 → `no`** — 수신 시각 기준 이미 지나간 시각의 이벤트. "어제 회의 잘 됐어요", "결제가 완료되었어요", 오늘 이미 지난 시각의 이벤트 등. **과거형 동사("~됐어요", "~완료되었어요", "~마쳤습니다") 포함.**

**Q1. (Q0이 '예') 사용자 본인이 갈/할 일인가?**
- **아니오 → `no`** — 거래·결제·적립·배송·인증·광고·인사·안부·남의 일정 등 "내가 직접 할 행동이 아닌" 모든 것. (단순 정보·정책 안내·남의 일정만 no. **내가 신청·참여할 수 있는 모집·공고·지원사업은 Q1=예 → pending**.)

**Q2. (Q1이 '예') 그게 확정됐는가? + 내일 이후인가?**
- **확정 + 내일 이후 → `yes`** — 내가 확실히 참석/수행하는 약속(시각 없는 종일 약속·"뵙겠습니다/하겠습니다" 단정형 포함). 내일 이후여야 자동등록 대상.
- **확정 + 오늘 미래 시각 → `pending`** — 오늘 중 아직 오지 않은 확정 일정. 앱이 예비로 표시(EventRouter: 오늘 yes → pending 강등).
- **미확정 → `pending`** — 안내·초대·제안·신청·마감 등 갈/할 수 있으나 아직 미정.

★ **날짜·시각·금액·기관명 같은 표면 토큰은 판단에 쓰지 않는다.** date는 yes·pending일 때만 추출하는 결과값이다(없으면 null — 지어내지 마라). no면 `events: []`.
★ **멀티턴**: Q2의 "확정"은 대화가 합의에 도달했는지로 본다 — 제안만 하고 끝 = pending, 거절·취소·모호("조만간 보자") = no.
★ **타인 일정 언급**: 발신자가 자신의 일정을 맥락으로 말하는 것("저 오늘 회의 있어서 나중에 연락드릴게요")은 Q1=아니오 → no. 사용자 일정이 아님.

## 대조 예시

| 메시지 | 수신 시각 | 라벨 | 판정 |
|---|---|---|---|
| 내일 3시 회의 | 오늘 오후 | **yes** | Q0 예·Q1 예·Q2 확정+내일이후 |
| 월요일 오후 2시에 뵙겠습니다 | 오늘 | **yes** | Q0 예·Q1 예·Q2 확정+미래 |
| [병원] 6/3 10:30 진료 예약 | 6/1 오전 | **yes** | Q0 예·Q1 예·Q2 확정+내일이후 |
| (스레드) "2시 어때요?" → "네 그때 봬요" | 이전 | **yes** | 대화 합의 도달 |
| 오늘 오후 3시 회의 | 오늘 오후 1시 | **pending** | Q0 예·Q1 예·Q2 확정+오늘미래 |
| 전문가 설명회 1/28 14시 안내 | 1/25 | **pending** | Q0 예·Q1 예·Q2 미확정 |
| 11일 실업인증 인터넷 신청 | 11일 오전 | **pending** | Q0 예·Q1 예(내가 신청)·Q2 미확정 |
| 예비창업패키지 참여기업 모집 ~7/10 | 7월 초 | **pending** | Q0 예·Q1 예(내가 신청 가능)·Q2 미확정 |
| 화요일 13시 어때요? (제안) | 월요일 | **pending** | 미수락 제안 |
| 오늘 오후 1시 회의 완료 | 오늘 오후 2시 | **no** | Q0 아니오 (이미 지남) |
| 결제가 완료되었어요 | 지금 | **no** | Q0 아니오 (과거형 완료) |
| 저 오늘 건축심의 회의 14시 있어 마치고 연락드릴게요 | 오늘 | **no** | Q1 아니오 (발신자 본인 일정) |
| 정책 6/17부터 변경 안내 | 6/15 | **no** | Q1 아니오 (단순 정보) |
| 어제 세미나 잘 마쳤습니다 | 오늘 | **no** | Q0 아니오 (과거) |
| 다른 일정 있어 다음에 | — | **no** | 거절 |

## 앱-모델 역할 분담

모델은 Q0~Q2 기준으로 분류·추출. 최종 등록 결정은 앱이 보정:

| 모델 출력 | 이벤트 시각 | 앱 처리 |
|---|---|---|
| yes | 내일 이후 | 자동등록 |
| yes | 오늘 미래 시각 | pending 강등 → 예비 알림 |
| yes | 현재 이전 (오류) | pending 강등 (EventRouter 안전망) |
| pending | 모든 경우 | 예비 알림 |
| no | — | 무시 |

## QA
각 판정에 `_label_reason` 한 줄(Q0/Q1/Q2 어디서 갈렸는지). generator 라벨 → evaluator 독립 재판정, 불일치 reject/fix. confidence 없음.

> 단서: "강의하셔야 하는 상황입니다"처럼 Q1이 진짜 애매한 경계는 남는다(0.6B 천장). 규칙으로 못 잡으니 다수를 깨끗이 가져가는 쪽을 택한다.

# 일정 양성/음성 판단 기준 (단일 SOT — generator·evaluator·golden 공용)

> 이 문서가 has_schedule 판단의 **유일한 기준**이다. generator(생성)·evaluator(QA)·golden(라벨)이 모두 이걸 따른다.
> 판단은 **두 갈래로 분리**한다 — A(단일 메시지)와 B(멀티턴 대화)는 서로 다른 기준을 쓴다.

---

## 0. 공통 원칙 — 사용자 행동 테스트

**양성(has_schedule=true) = 이 메시지로 수신자(사용자) 본인에게, 특정 일시에 직접 참석·수행할 약속/일정이 생기는가?**
캘린더에 넣으면 그 시각에 사용자가 **실제로 가거나 무언가를 하게** 되는 것.

★ **날짜·시각의 존재는 판단 근거가 아니다.**
- "적립일시 6/15 10:43", "결제예정일 6/17", "승인 12:10", "배송 예정 6/17"은 날짜·시각이 있어도 **음성**.
- "끝나고 보자"(시각 없음)도 약속이면 **양성**.
- 순서: **① 사용자의 약속인가(양/음 판단) → ② 양성이면 date/time을 정확히 추출.** 추출과 판단은 분리.

★ 모든 페어/판정은 `_label_reason`(왜 양/음성인지 한 줄)을 함께 남긴다(감사·일관성용).

---

## A. 단일 메시지 기준 — "내용의 종류"로 판단

자기완결적 메시지 하나. **약속을 알리는 메시지인가, 아니면 통보/정보인가**로 분류.

### A-양성: 확정 일정 알림 (★ 매우 구체적으로 탐지)

수신자가 **특정 일시에 참석·수행할 약속**이 메시지 하나로 확정·통보됨. 두 갈래:

**(1) 개인 통보** — 지인·동료가 약속을 알림/잡음
- "내일 3시 강남역 6번출구에서 봐요", "내일 12시 외할머니댁, 저녁 7시 이모네 모임"

**(2) 기관/서비스 예약·확정 알림** — 아래 유형을 적극적으로 양성 탐지:
- **의료**: "[○○병원] 6/3(수) 10:30 진료 예약 안내" / 치과·검진·상담 예약
- **항공·철도·버스**: "KE093 6/14 18:30 ICN 출발 확정" / "SRT 6/20 09:00 예약 완료"
- **숙박·식당**: "○○호텔 6/20 15:00 체크인" / "○○ 6/18 19:00 예약 확정"
- **면접·채용**: "1차 면접 6/16(화) 14:00 ○○빌딩 3층"
- **교육·시험**: "오리엔테이션 6/15 10:00 ○○관" / 수업·시험 일정
- **방문 서비스**: "기사님 6/17 오전 10~12시 방문 예정"(설치/AS — 사용자가 대기·응대)
- **행사(설명회·포럼·세미나·간담회·출범식·공유회·총회·교육)**: ★ **명시된 일시(날짜+시각)에 열리고 사용자가 참석 대상**이면 양성. **"참석 안내/요청/개최 안내"여도 일시가 정해졌으면 양성.** (예: "전문가 대상 설명회 1/28(수) 14시", "인스트럭터 포럼 11/2 16:00", "성과공유회 12/9 참석 요청", "○○ 출범식 6/16 14:00")

판별 테스트: **그 시각에 사용자가 직접 가거나, 그 자리에 있어야 하거나, 능동적으로 무언가 해야 하는가?** → yes면 양성.
추출: 이런 알림은 대개 **절대 날짜·시각·장소가 명시** → 정확히 추출, confidence 높게.

### A-음성: 비-일정 통보 (★ 날짜·시각 있어도 음성)

사용자가 **그 시각에 능동적으로 행동하지 않음**(자동 발생/수동 수신/단순 정보):
- **거래/금융**: 카드 승인·출금·입금·결제 예정·정기결제·청구서
- **적립/포인트/멤버십/구독/쿠폰**: "적립일시 …", "사용기한 …", "별 소멸 …"
- **배송/물류**: 배송 시작·도착 예정 (단순 수령, 사용자가 가는 게 아님)
- **인증/보안**: 로그인 시도·인증 유효시각
- **영업/상담 로지스틱스**: "오후에 연락드릴게요", "견적/사진 보내드릴게요", "검토 후 회신"
- **광고/홍보/이벤트 안내**, **인사말·안부·잡담**
- **마감/공고/신청/납부**: 모집공고·지원서 접수마감·국가장학금/세금 납부·신고·건강검진 안내·도서 반납 등 — **고정 참석 일시가 없고 기한 내 아무때나 하는 task** → 음성. (예: "국가장학금 신청 마감 6/19", "등록금 납부 7일", "건강검진 12/31까지", "SW마에스트로 모집 ~6/10", "교육 모집 ~11/30")
- **상업적 조인/구독/멤버십 요청 알림**: 서비스가 보내는 미수락 권유(고정 참석 약속 아님) → 음성. (예: "골프조인 요청서 도착 6/27 05:40", "선착순 마감 신청", 광고성 행사 권유). ★ **단 기관이 사용자를 참석 대상으로 초대한 업무 행사(설명회·포럼 등, 일시 명시)는 위 A-양성** — '모집/조인 요청'과 '일시 정해진 행사 참석'을 구분하라.
- **단순 정보·뉴스·시행일**: "6/17부터 정책 변경"
- **과거 이벤트 회고**: "어제 14시 세미나 잘 마쳤습니다"
- **남의 일정 언급**: "내일 종일 간담회라 정신없을 듯"(사용자 약속 아님)

### A-경계 (명확히)
- 배송 **"방문 수령"** 명시 → 양성 / 단순 도착 통보 → 음성.
- **마감/신청/납부/공고 "~까지"** → **음성**(특정 시각 참석 아님, 기한 task). ※ 단 사용자가 그 일시에 **직접 가야 하는** 행사·예약·면접·방문은 마감 표현이 있어도 양성.
- **방문 서비스/예약 알림**(가전점검·모뎀교체·치과/병원 예약 "○월○일 ○시 가능 여부 확인/예약 안내") → **양성**(사용자가 그 일시에 응대·방문). 단 **미수락 초대/조인/광고**는 위처럼 음성.

---

## B. 멀티턴(대화) 기준 — "확정 도달 여부"로 판단

여러 메시지의 흐름. **대화가 약속에 합의했는가**로 분류. 확정은 메시지 내용이 아니라 **대화 상태**에서 나온다.

### B-양성: 대화로 확정됨
- 마지막 상태가 **확정**: 양쪽이 특정 날짜·시각에 **합의**했고 마지막이 그 수락/재확인.
- 확정 신호: "좋습니다 / OK / 네 알겠습니다 / 그렇게 하시죠 / 네 그럼 화요일 13시에 봬요 / 내일 뵙겠습니다 / Confirmed"
- ★ **하나의 약속 = 한 건.** 협의 과정의 여러 제안은 별도 일정이 아니다.
- ★ **최종 합의값 추출** — 중간에 바뀐 제안 말고 **마지막으로 합의된** 날짜·시각·장소.
  - 예: "월요일 어때요? → 아니 화요일 13시 → 네 화요일 13시로" → **화요일 13시**.
- **재확인/리마인드/링크 공유**("화요일 13시 줌 방 공유합니다", "내일 뵈요")도 **이미 확정된 그 약속**을 가리킴 → 그 약속을 양성으로 추출(새 일정·중복 아님).

### B-음성: 미확정
- **새 제안/질문으로 끝남**: "3시 어때요?", "화요일 가능하세요?" → 아직 확정 아님.
- **유보/모호**: "조만간 보자", "다음 주에 잡을게요"(구체 시각 없음).
- **거절/취소**: "다른 일정 있어 다음에", "회의 취소합니다".
- 협의 무관 잡담.

### B-판별 절차
1. 마지막 메시지가 **확정 수락**인가? (제안/질문이면 → 음성)
2. 합의된 **구체 날짜·시각**이 있는가? (모호하면 → 음성)
3. **거절·취소·유보**인가? (맞으면 → 음성)
→ (1 yes) & (2 yes) & (3 no) → **양성**, 최종 합의값 추출.

## 대조 예시 (같은 표면형, 반대 라벨 — 토큰 의존 차단)

| 메시지 | 라벨 | 이유 |
|---|---|---|
| "내일 14시 강남역에서 봐요" | 양성(A) | 사용자가 그 시각에 감 |
| "[KB] 06/15 14:32 출금 50,000원" | 음성(A) | 자동 거래 통보 |
| "[○○병원] 06.03 10:30 진료 예약 안내" | 양성(A) | 사용자가 병원 방문 |
| "카카오페이 적립 6/15 10:43, 사용기한 2028.6.30" | 음성(A) | 적립 통보, 행동 없음 |
| "6/17 14:00 간담회 참석 부탁드립니다" | 양성(A) | 사용자 참석 |
| "Buddy Pass 정기결제 예정일 6/17" | 음성(A) | 자동 결제 |
| "…화요일 13시 줌으로 봬요"(확정) | 양성(B) | 대화 확정, 최종 합의 |
| "화요일 13시 가능하세요?"(질문으로 끝) | 음성(B) | 미확정 제안 |

## 출력/QA 규약
- 생성·라벨 시 각 페어에 `_label_reason` 한 줄(위 기준의 어느 항목인지).
- **이중 합의**: generator가 라벨 → evaluator가 **독립 재판정**, 불일치는 reject/fix(기준 일관성 강제).
- 신뢰도(confidence): "날짜+시간 있으면 0.9" 같은 **토큰 공식 폐기**. **확실한 약속일수록 높게**(약속 명확+시점 분명 0.9+, 시점 모호/마감형 0.6~0.8).

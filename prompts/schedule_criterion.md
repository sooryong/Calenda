# 일정 분류 기준 (SOT — generator·evaluator·golden·audit 공용)

`schedule_status` ∈ `"yes" | "pending" | "no"`. 아래 **두 질문만으로** 분류한다.

| 값 | 의미 | events | 앱 |
|---|---|---|---|
| **yes** | 확정 일정 | 추출 | 자동등록 |
| **pending** | 미확정 가능성 | 추출 | 예비 |
| **no** | 비-일정 | `[]` | 무시 |

## 분류 테스트 (이게 본문 전부)

**Q1. 사용자 본인이 갈/할 미래의 일인가?**
- **아니오 → `no`** — 거래·결제·적립·배송·인증·광고·인사·안부·과거 회고·남의 일정 등 "내가 직접 할 행동이 아닌" 모든 것. (단순 정보·정책 안내·남의 일정만 no. **내가 신청·참여할 수 있는 모집·공고·지원사업은 Q1=예 → pending**.)

**Q2. (Q1이 '예') 그게 확정됐는가?**
- **확정 → `yes`** — 내가 확실히 참석/수행하는 약속(시각 없는 종일 약속·"뵙겠습니다/하겠습니다" 단정형 포함).
- **미확정 → `pending`** — 안내·초대·제안·신청·마감 등 갈/할 수 있으나 아직 미정.

★ **날짜·시각·금액·기관명 같은 표면 토큰은 판단에 쓰지 않는다.** date는 yes·pending일 때만 추출하는 결과값이다(없으면 null — 지어내지 마라). no면 `events: []`.
★ **멀티턴**: Q2의 "확정"은 대화가 합의에 도달했는지로 본다 — 제안만 하고 끝 = pending, 거절·취소·모호("조만간 보자") = no.

## 대조 예시 (날짜 있는 것·없는 것을 섞음 — date 무관 학습)
| 메시지 | 라벨 | 판정 |
|---|---|---|
| 내일 3시 회의 | **yes** | Q1 예 · Q2 확정 |
| 월요일 오후 2시에 뵙겠습니다 | **yes** | Q1 예 · Q2 확정(약속 단정) |
| [병원] 6/3 10:30 진료 예약 | **yes** | Q1 예 · Q2 확정 |
| (스레드) "2시 어때요?" → "네 그때 봬요" | **yes** | 대화 합의 도달 |
| 전문가 설명회 1/28 14시 안내 | **pending** | Q1 예 · Q2 미확정 |
| 장학금 신청해야 하는데 (날짜 없음) | **pending** | Q1 예 · Q2 미확정 |
| 11일 실업인증 인터넷 신청 | **pending** | Q1 예(내가 신청) · Q2 미확정 |
| 예비창업패키지 참여기업 모집 ~7/10 | **pending** | Q1 예(내가 신청 가능) · Q2 미확정 |
| 화요일 13시 어때요? (제안) | **pending** | 미수락 제안 |
| 정책 6/17부터 변경 안내 | **no** | Q1 아니오 (단순 정보) |
| Buddy Pass 결제예정 6/17 | **no** | Q1 아니오 (자동 거래) |
| 어제 세미나 잘 마쳤습니다 | **no** | Q1 아니오 (과거) |
| 다른 일정 있어 다음에 | **no** | 거절 |

## QA
각 판정에 `_label_reason` 한 줄(Q1/Q2 어디서 갈렸는지). generator 라벨 → evaluator 독립 재판정, 불일치 reject/fix. confidence 없음.

> 단서: "강의하셔야 하는 상황입니다"처럼 Q1이 진짜 애매한 경계는 남는다(0.6B 천장). 규칙으로 못 잡으니 다수를 깨끗이 가져가는 쪽을 택한다.

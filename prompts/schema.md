# 일정 추출 출력 JSON 스키마

Planner / Generator / Evaluator 및 학습 데이터 전체가 공유하는 단일 출력 스키마.

## 최상위 객체

```json
{
  "has_schedule": true,
  "events": [
    {
      "title": "팀 주간 회의",
      "start": "2026-05-26T15:00:00+09:00",
      "end":   "2026-05-26T16:00:00+09:00",
      "all_day": false,
      "location": "회사 3층 회의실",
      "attendees": ["김부장", "박과장"],
      "description": null,
      "recurrence": null,
      "confidence": 0.95
    }
  ]
}
```

## 필드 정의

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `has_schedule` | bool | ✅ | 메시지에 등록 가치 있는 일정이 있는지 |
| `events` | array | ✅ | 일정 객체 배열. `has_schedule=false`면 `[]` |
| `events[].title` | string | ✅ | 일정 제목. 메시지에서 자연스럽게 추출/요약 |
| `events[].start` | ISO8601 string \| null | ✅ | 시작 시각. 시간대 포함(`+09:00` for KST) |
| `events[].end` | ISO8601 string \| null | ⛔ optional | 끝 시각. 명시 없으면 null |
| `events[].all_day` | bool | ✅ | 종일 일정 여부 |
| `events[].location` | string \| null | ⛔ | 장소. 없으면 null |
| `events[].attendees` | string[] | ⛔ | 참석자 이름/별칭. 없으면 [] |
| `events[].description` | string \| null | ⛔ | 부가 설명 |
| `events[].recurrence` | string \| null | ⛔ | RRULE 형식. 명시적 반복일 때만. |
| `events[].confidence` | float 0~1 | ✅ | 추출 자신도 |

## 규칙

1. 출력은 **순수 JSON**이어야 함. 앞뒤 설명문/마크다운 코드펜스 금지.
2. 메시지에 명시되지 않은 정보는 절대 환각하지 않음 → `null`.
3. 상대 시간("내일", "다음 주 화요일")은 입력에 제공된 `<수신시각>`을 기준으로 절대 시각으로 변환.
4. 시간대가 메시지에 명시 안 되면 **Asia/Seoul (+09:00)** 가정.
5. 일정이 없으면 `{"has_schedule": false, "events": []}`만 반환.
6. 한 메시지에 여러 일정이 있으면 `events` 배열에 모두 포함.
7. 시간만 있고 날짜가 명시 안 된 경우, 가장 가까운 미래 날짜로 해석 (예: "오늘 3시"가 이미 지났으면 내일 3시가 아니라 그대로 오늘 3시로).
8. **past_time**: 메시지에 명시된 일정이 `received_at` 기준으로 이미 지난 시각이어도 `has_schedule=true`로 라벨링하고 `start`는 그 과거 시각 그대로 기록한다. 실제 캘린더 등록 여부는 다운스트림(앱 로직)이 판단한다. 모델은 "일정 추출"만 책임진다.
9. **반복 일정**: "매주 화요일 회의", "매월 1일 결제" 같이 명시적 반복이면 `has_schedule=true`로 보고 `recurrence`에 RRULE 작성, `start`에는 첫 회 시작 시각.

## 필드 가이드라인

### `title`
- 메시지에서 핵심 요약. 발신자 소속이 사실상 일정의 정체성이면 포함 (예: "서울내과 진료", "Korean Air KE093").

### `location`
- 장소가 명시된 경우만. 발신자 소속이 곧 장소면 발신자명을 location에 사용 가능 (예: 병원/식당 예약).

### `attendees`
- 메시지에 등장하는 **참석자**의 이름/별칭. 발신자(`<발신자:>`) 자신은 보통 포함하지 않음 (단, "팀장님이 호스트"처럼 발신자가 명시적 참석자로 언급되면 포함).

### `description`
- 부가 메모. 다음 정보들이 들어간다: **전화번호, 이메일, URL, 변경/취소 안내, 준비물, 드레스코드, 항공편 메모** 등.
- 메시지에서 직접 인용하거나 짧게 요약. 메시지에 없는 정보는 절대 추가 금지.

### `recurrence`
- RRULE 표기. 예: `FREQ=WEEKLY;BYDAY=TU` (매주 화요일), `FREQ=MONTHLY;BYMONTHDAY=1`.
- 일회성 일정이면 `null`.

### `confidence` (모델·gold 공통 스케일)
| 범위 | 케이스 |
|---|---|
| **0.90 ~ 1.0** | 날짜·시간·장소 모두 명시된 확정 일정 |
| **0.75 ~ 0.90** | 핵심 정보(날짜+시간) 있고 부가 정보 일부 누락 |
| **0.60 ~ 0.75** | 상대 시간만 있거나 장소 모호, 추론 일부 필요 |
| **0.40 ~ 0.60** | 부분 정보만 있고 추론 비중 큼 (시간만 또는 날짜만) |
| **`has_schedule: false`** | confidence 필드 자체 생략 또는 N/A — events가 빈 배열이므로 의미 없음 |

## Title 표기 규칙 (캘린더 등록 형식)

캘린더 일정 카드에 `[제목, 장소: 발신자(소속)]` 형식으로 보이도록 title 필드를 작성:

| 발신자 유형 | title 형식 |
|---|---|
| **외부 개인** (카톡/문자) | `"{제목}, {장소}: {발신자}"` |
| **외부 개인** (gmail, 이메일 발신) | `"{제목}, {장소}: {이름} ({도메인소속})"` 예: "Project Kickoff Meeting, Conf Room A: Sarah Lee (Company)" |
| **외부 기관** (병원/항공사/관리사무소) | `"{제목}, {장소}: {기관명}"` 발신자=기관이므로 소속 괄호 생략 |
| **본인 발신** (`sender: "나"` 또는 `"Me"`) | `"{제목}, {장소}"` 발신자 정보 생략 |
| **발신자=장소** (예: 발신=서울내과, 장소=서울내과) | 한쪽 생략: `"{제목}: {기관명}"` |
| **제목에 이미 발신자·장소 포함** | 그대로 두고 끝에 `": {발신자}"` 추가 |

### Title 작성 룰

- 장소(`location`) 정보를 title에 명시적으로 포함시켜 캘린더 일별 보기에서 바로 인지되도록.
- `location` 필드는 별도로 그대로 채움 (title 안에 있어도 중복 저장 OK — Google Calendar는 location을 별도 처리).
- 발신자 식별:
  - 카톡/문자: `<발신자>` 토큰 그대로 사용
  - Gmail: `name@domain.com` → 이름 추출 (메시지 본문에 명시되면 그것 우선), 도메인 → 회사명 (예: `@company.com` → `Company`)
  - 본인 식별: `sender: "나"` 또는 본인 이메일 (`sooryong.byun@gmail.com` = 변수룡 본인)

### 데이터 페어 구조 (학습 데이터 형식)

JSONL 한 줄에 다음 객체:

```json
{
  "scenario_id": "string",
  "received_at": "ISO8601",
  "channel": "kakao|sms|gmail",
  "sender": "string",
  "language": "ko|en|mixed_ko_en",
  "thread_context": [               // optional, 멀티턴 케이스에만
    {"time": "HH:MM", "sender": "string", "message": "string"},
    ...
  ],
  "message": "string",
  "gold": { "has_schedule": ..., "events": [...] }
}
```

`thread_context`가 없거나 빈 배열이면 단일 메시지 케이스. 있으면 멀티턴.

## 입력 포맷 (학습/추론 시)

### 기본 (단일 메시지)

```
<채널: KakaoTalk>
<수신시각: 2026-05-25T14:30:00+09:00>
<발신자: 김부장>
<메시지>
내일 3시에 회사 3층에서 주간회의 잡았습니다. 박과장도 같이.
</메시지>
```

### 확장 (멀티턴 — 대화내역 포함)

카톡/문자/Gmail의 일정 협의가 여러 메시지에 걸쳐 진행되는 경우, 안드로이드 앱이 직전 3~5개 메시지를 함께 입력:

```
<채널: KakaoTalk>
<수신시각: 2026-05-25T10:35:00+09:00 (월)>
<발신자: 나>
<대화내역>
[10:02] 김용안: 줌으로 한다면, 언제 가능하실까요?
[10:04] 나: 29일 오전, 30일 토 오전, 6/1 종일
[10:07] 나: 30일 토요일 오전에 줌으로 할까요?
[10:14] 김용안: 6월4일 오전 9시 어떤지요?
</대화내역>
<메시지>
좋습니다
</메시지>
```

### 입력 포맷 규칙

- `<대화내역>`은 **optional**. 단일 메시지로 의미가 명확하면 생략 (블록 자체를 넣지 않음).
- 대화내역 표기 형식: `[HH:MM] 발신자: 메시지` 한 줄씩. 시간순 (오래된 것부터).
  - ★ 이 형식은 `scripts/_common.py:build_user_block`이 학습·평가·(추후 앱) 공용으로 렌더링한다. 형식을 바꾸면 세 곳을 함께 바꿔야 함 (train/추론 분포 불일치 주의).
- `<수신시각>`에는 한국어 요일 `(월)`을 부착 (단일/멀티턴 동일).
- 보통 직전 3~5개. 너무 길면 모델 부담 (300토큰 권장).
- 본인 발신자는 `나` (ko/mixed) 또는 `Me` (en).
- 최종 메시지(가장 아래 `<메시지>`)가 모델이 일정 추출을 시도하는 **대상**. 대화내역은 그 메시지를 해석하기 위한 참고.

### 모델 판단 가이드 (멀티턴)

- 최종 메시지가 **확정 응답** (예: "좋습니다", "OK", "그렇게 하시죠", "네 알겠습니다")이면 스레드 맥락에서 직전 제안된 시각·장소를 추출해서 `has_schedule=true`.
- 최종 메시지가 **새 제안**이면 `has_schedule=false` (다음 확정 응답을 기다림).
- 스레드 맥락에 여러 시각 후보가 나왔다면 **가장 최근 제안된 것**이 최종 합의 시각.
- 최종 메시지가 **취소·변경 통보**면 스레드 맥락의 기존 합의와 비교해서 변경/취소 처리.

→ 모델은 위 스키마 JSON을 출력해야 함.

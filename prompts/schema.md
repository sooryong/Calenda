# 일정 추출 출력 JSON 스키마

Planner / Generator / Evaluator 및 학습 데이터 전체가 공유하는 단일 출력 스키마.

## 설계 원칙

**모델은 텍스트 표면형을 추출만 하고, 날짜·시각 산술은 앱/eval의 resolver가 결정론적으로 수행한다.**

모든 필드는 **동등한 수준**에서 독립 추출한다:
- `is_schedule` = "이 메시지가 나의 확정 일정인가?" (binary 판단)
- `title / date / time / location / description` = 텍스트에서 추출 (`is_schedule` 값과 무관하게 있는 것만)

단일 이벤트만 처리한다 (멀티이벤트 미지원).

## 출력 형식

```json
{
  "is_schedule": true,
  "title": "AWS 교육팀 줌회의",
  "date": "내일",
  "time": { "hour": 1, "minute": 0, "marker": "오후" },
  "end_time": null,
  "location": "줌",
  "description": null
}
```

## 필드 정의

| 필드 | 타입 | 설명 |
|------|------|------|
| `is_schedule` | `true`\|`false` | **나의 확정 일정**이면 `true`. 거래·통보·광고·미수락 제안·행사공고·단순 안내이면 `false`. ★ 분류 기준 = `prompts/schedule_criterion.md` |
| `title` | string \| null | 일정 제목. 메시지의 주제 문구를 시간 표현만 제외하고 최대한 그대로 보존. 정보가 없으면 null |
| `date` | string \| null | **상대 토큰** (`내일`, `다음주화`, `1주후`…) 또는 **절대일자** (`"2026-06-05"`) 또는 null. 모델은 절대일자 계산 금지 — 텍스트 표면형 그대로 토큰화. |
| `time` | object \| null | `{hour, minute, marker}` 또는 null(시각 미상·종일). null이면 앱이 자동으로 종일 처리. 24h 변환은 resolver가 함. |
| `end_time` | object \| null | 종료 시각. `{hour, minute, marker}` 또는 null |
| `location` | string \| null | 물리 장소 또는 온라인 도구(`줌`, `구글밋`, `팀즈`). 없으면 null |
| `description` | string \| null | 부가 정보 통합 필드. 참석자·주최자·반복 일정·전화번호·URL·준비물 등. 형식 예: `"참석자: 박과장, 이대리\n반복: 매주 화요일"`. 없으면 null |

### `time` / `end_time` 객체

| 키 | 타입 | 설명 |
|---|---|---|
| `hour` | int | 시 숫자. 1~12(표시어와 함께) 또는 0~23(24h 표기) |
| `minute` | int | 분. 기본 0 |
| `marker` | string \| null | `오전`·`오후`·`저녁`·`밤`·`낮`·`새벽`·`아침`·`정오`·`자정` 또는 null |

## 규칙

1. 출력은 **순수 JSON**. 앞뒤 설명문·마크다운 코드펜스 금지.
2. 메시지에 없는 정보는 절대 환각하지 않음 → `null`.
   - 시각이 없으면 `time: null`, 장소가 없으면 `location: null`, 제목이 없으면 `title: null`.
3. **날짜·시각은 계산하지 말고 추출만.** 상대 날짜는 아래 어휘표의 토큰으로, 시각은 `{hour, minute, marker}`로.
4. `is_schedule=false`여도 날짜·장소·제목 등 추출 가능한 필드는 채운다. 정보가 없는 필드만 null.
5. `date`가 null이고 `time`만 있으면 resolver가 **오늘**로 가정.
6. **수신 시각 기준**: 이미 지난 이벤트·완료 알림이면 `is_schedule=false`.

## 날짜 토큰 어휘 (R = 수신 날짜)

| 텍스트 | 토큰 | resolver |
|---|---|---|
| 오늘 / 내일 / 모레 / 글피 | `오늘` `내일` `모레` `글피` | R+0/+1/+2/+3일 |
| N일 뒤/후 | `N일후` | R+N일 |
| 1주일 뒤 / 다음 주(요일無) | `1주후` | R+7일 |
| N주 뒤 | `N주후` | R+7N일 |
| 한 달 / N개월 뒤 | `N개월후` | R+N개월 |
| 이번/다음/다다음 주 X요일 | `이번주화` `다음주화` `다다음주화` | 해당 주의 X요일 |
| 이번/다음 주말 | `이번주말` `다음주말` | 다가오는 토요일 |
| N일(단독) | `30일` `15일` | 가까운 미래 N일 |
| 6월 5일 / 6/5 | `6월5일` `6/5` 또는 `2026-06-05` | 가까운 미래(월-일) 또는 절대 ISO |
| 날짜 없음 | `null` | (time만 있으면 오늘) |

## time 변환 (marker → 24h)

| marker | 규칙 | 예 |
|---|---|---|
| 오후·저녁·밤·낮 | +12 (12시→12 유지) | 오후 3시 → 15:00 |
| 오전·아침·새벽 | 그대로 (12시→00) | 오전 9시 → 09:00 |
| 정오 / 자정 | 12:00 / 00:00 | |
| null + hour≥13 | 그대로 (24h) | 15시 → 15:00 |
| null + hour 1~12 | 받은 시각 이후 가장 가까운 쪽 | |

## 필드 가이드라인

### `title`
- 메시지의 일정 제목/주제를 **시간 표현만 제외하고 원문 그대로 최대한 보존**. 절대 임의 축약·재해석 금지.
- "내일 13시 AWS 교육팀과 줌회의" → `AWS 교육팀과 줌회의`
- "2025년 전자부품산업기술개발(주력산업IT융합) 신규과제 선정평가 관련 안내" → `전자부품산업기술개발(주력산업IT융합) 신규과제 선정평가`
- "RISE사업 의료바이오 특화분야 창업패키지 선정평가(발표)" → `RISE 의료바이오 창업패키지 선정평가(발표)`
- ❌ 금지: 긴 제목을 짧게 줄이거나, 메시지의 다른 부분(장소·발신자·날짜)을 제목으로 사용
- `is_schedule=false`여도 메시지 주제를 제목으로 추출. 예: "초기창업패키지 모집 안내" → `title: "초기창업패키지 모집 안내"`
- 멀티턴: 이전 메시지의 활동명을 제목으로 추출. 장소·발신자 이름을 제목으로 복사하는 것 금지.

### `location`
- 물리 장소(`회사 3층`, `강남역 2번출구`) 또는 온라인 도구(`줌`, `구글밋`, `팀즈`).
- 제목을 location에 복제하지 말 것. 장소가 없으면 `null`.

### `description`
- 참석자, 주최자, 반복일정, URL, 전화번호, 준비물 등 부가 정보를 한 필드에 통합.
- 형식 예:
  ```
  참석자: 박과장, 이대리
  주최: 대구창조경제혁신센터
  반복: 매주 화요일
  ```
- 메시지에 없는 정보는 추가 금지.

## is_schedule 판단 기준

| 메시지 유형 | is_schedule |
|---|---|
| 내가 참여하는 확정 약속·회의·일정 | `true` |
| 협의 완료("네", "좋습니다") → 직전 제안 시각 추출 | `true` |
| 행사 공고·모집 안내·초대(참석 미확정) | `false` |
| 결제·거래·배송 알림 | `false` |
| 새 제안·협의 중·유보 | `false` |
| 이미 지난 이벤트·완료 알림 | `false` |

## 데이터 페어 구조 (학습 데이터 형식)

```json
{
  "scenario_id": "string",
  "received_at": "ISO8601",
  "channel": "kakao|sms|gmail",
  "sender": "string",
  "language": "ko|en|mixed_ko_en",
  "thread_context": [
    {"time": "HH:MM", "sender": "string", "message": "string"}
  ],
  "message": "string",
  "gold": {
    "is_schedule": true,
    "title": "AWS 교육팀 줌회의",
    "date": "내일",
    "time": {"hour": 1, "minute": 0, "marker": "오후"},
    "end_time": null,
    "location": "줌",
    "description": null
  }
}
```

## 입력 포맷 (학습/추론)

### 단일 메시지
```
<채널: KakaoTalk>
<수신시각: 2026-05-25T14:30:00+09:00 (월)>
<발신자: 김부장>
<메시지>
내일 오후 1시에 줌으로 AWS 교육팀과 회의합니다.
</메시지>
```

### 멀티턴 (대화내역 포함)
```
<채널: KakaoTalk>
<수신시각: 2026-05-25T10:35:00+09:00 (월)>
<발신자: 나>
<대화내역>
[10:14] 김용안: 6월4일 오전 9시 어떤지요?
</대화내역>
<메시지>
좋습니다
</메시지>
```

### 멀티턴 판단 규칙
- 최종 메시지가 확정 응답("좋습니다", "OK", "네") → 직전 제안 시각 추출, `is_schedule=true`
- 새 제안·유보·거절 → `is_schedule=false`
- 여러 시각 후보 → 가장 최근 제안된 것

## Title 표시 규칙

모델 `title` = 메시지의 자연 제목(시간만 제외). 앱 `compose_title`은 발신인 태그만 추가:

표시 제목 = `{모델 title} [{발신인(소속)}]`

# 일정 추출 출력 JSON 스키마

Planner / Generator / Evaluator 및 학습 데이터 전체가 공유하는 단일 출력 스키마.

## 설계 원칙 (중요)

**모델은 텍스트 표면형을 추출만 하고, 모든 날짜·시각 산술은 앱/eval의 resolver가 결정론적으로 수행한다.**
0.5B는 "요일→날짜"(주말 횡단)·"AM/PM→24h" 같은 산술을 못 맞히므로, 모델에겐 추출(잘하는 일)만 시키고
계산(코드가 잘하는 일)은 `scripts/_common.resolve_when`(= 앱 `DateResolver`)에 맡긴다.
→ 모델은 `start`(절대 ISO)를 출력하지 않는다. 대신 **`date`(상대 토큰/절대일자)** 와 **`time`(시·분·표시어)** 을 낸다.

## 최상위 객체

```json
{
  "has_schedule": true,
  "events": [
    {
      "title": "정원구 대표 줌 멘토링",
      "date": "내일",
      "time": { "hour": 2, "minute": 0, "marker": "오후" },
      "end_time": null,
      "all_day": false,
      "location": null,
      "attendees": [],
      "organizer": null,
      "description": "줌(온라인)",
      "recurrence": null,
      "confidence": 0.9
    }
  ]
}
```

## 필드 정의

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `has_schedule` | bool | ✅ | 메시지에 등록 가치 있는 일정이 있는지 |
| `events` | array | ✅ | 일정 객체 배열. `has_schedule=false`면 `[]` |
| `events[].title` | string | ✅ | 일정 제목. 메시지에서 자연스럽게 추출/요약 |
| `events[].date` | string \| null | ✅ | **상대 토큰**(`내일`,`다음주화`,`1주후`…) 또는 **절대일자**(`"2026-06-05"`) 또는 null. ★모델은 텍스트 표현을 그대로 토큰화만, 절대일자 **계산 금지** (절대일자가 텍스트에 명시된 경우만 YYYY-MM-DD). 어휘는 아래 표. |
| `events[].time` | object \| null | ✅ | `{hour, minute, marker}` 또는 null(시각 미상/종일). 24h 변환은 앱이 함. |
| `events[].end_time` | object \| null | ⛔ | 종료 시각(같은 날). `{hour,minute,marker}` 또는 null. |
| `events[].all_day` | bool | ✅ | 종일 일정 여부 (true면 `time=null`) |
| `events[].location` | string \| null | ⛔ | 장소. 없으면 null |
| `events[].attendees` | string[] | ⛔ | 참석자 이름/별칭. 없으면 [] |
| `events[].organizer` | string \| null | ⛔ | **발신 기관/소속** (예: `서울내과`, `Lufthansa`). 메시지에 명시된 경우만, 없으면 null |
| `events[].description` | string \| null | ⛔ | 부가 설명 |
| `events[].recurrence` | string \| null | ⛔ | RRULE 형식. 명시적 반복일 때만. |
| `events[].confidence` | float 0~1 | ✅ | 추출 자신도 |

### `time` / `end_time` 객체
| 키 | 타입 | 설명 |
|---|---|---|
| `hour` | int | 텍스트의 시 숫자. 1~12(표시어와 함께) 또는 0~23(24h 표기) |
| `minute` | int | 분. 기본 0 |
| `marker` | string \| null | `오전`·`오후`·`저녁`·`밤`·`낮`·`새벽`·`아침`·`정오`·`자정` 또는 null(표시어 없음) |

## 규칙

1. 출력은 **순수 JSON**이어야 함. 앞뒤 설명문/마크다운 코드펜스 금지.
2. 메시지에 명시되지 않은 정보는 절대 환각하지 않음 → `null`.
3. **날짜·시각은 계산하지 말고 추출만.** 상대 날짜("내일","다음 주 화요일")는 아래 어휘의 `date` 토큰으로, 시각은 `time={hour,minute,marker}`로. 요일→날짜, AM/PM→24h 같은 산술은 전부 resolver(코드)가 함.
4. 시간대는 resolver가 **Asia/Seoul (+09:00)** 부여 (모델은 신경 안 씀).
5. 일정이 없으면 `{"has_schedule": false, "events": []}`만 반환.
6. 한 메시지에 여러 일정이 있으면 `events` 배열에 모두 포함.
7. `date`가 null이고 `time`만 있으면("3시에 보자") → resolver가 **오늘**로 가정.
8. **past_time**: 이미 지난 시각이어도 `has_schedule=true`로 라벨링하고 `date`/`time`은 그대로 기록. 캘린더 등록 여부는 앱이 판단. 모델은 "추출"만 책임진다.
9. **반복 일정**: "매주 화요일", "매월 1일" 같이 명시적 반복이면 `recurrence`에 RRULE, `date`에는 첫 회 날짜 토큰.

## 날짜·시각 토큰 어휘 (모델 추출 → resolver 변환)

`scripts/_common.resolve_when`(= 앱 `DateResolver`)가 아래 표대로 `received_at` 기준 절대 시각으로 변환한다.
**학습·평가·앱이 같은 표를 공유**하며, 표가 그대로 단위 테스트가 된다.

### `date` 토큰 (R = 수신 날짜)
| 텍스트 | 토큰 | resolver |
|---|---|---|
| 오늘 / 내일 / 모레 / 글피 | `오늘` `내일` `모레` `글피` | R+0/+1/+2/+3일 |
| N일 뒤/후 | `N일후` | R+N일 |
| 1주일 뒤 / 다음 주(요일無) | `1주후` | R+7일 |
| N주 뒤 | `N주후` | R+7N일 |
| 한 달 / N개월 뒤 | `N개월후` | R+N개월(월말 클램프) |
| 이번/다음/다다음 주 X요일 | `이번주화` `다음주화` `다다음주화` | 해당 주의 X요일 (월요일 시작 주) |
| 이번/다음 주말 | `이번주말` `다음주말` | 다가오는 토요일(+7) |
| N일(월 없이 단독) | `30일` `15일` | 가까운 미래 N일(이번달 N일이 R 이후면 이번달, 아니면 다음달; 월말 클램프) |
| 6월 5일 / 6/5 | `6월5일` `6/5` 또는 `2026-06-05` | 월-일 토큰=가까운 미래(연도 추론, 월말 클램프). 절대 ISO는 그대로. ★모델은 표면형 토큰 권장(연도 산술 회피) |
| 날짜 없음 | `null` | (time만 있으면 오늘 가정) |

### `time` 변환 (marker → 24h)
| marker | 규칙 | 예 |
|---|---|---|
| 오전·아침·새벽 | 그대로 (12시→00) | 오전 9시 → 09:00 |
| 오후·저녁·밤·낮 | +12 (12시→12 유지) | 저녁 8시 → 20:00 |
| 정오 / 자정 | 12:00 / 00:00 | |
| `null` + hour≥13 | 그대로 (24h 표기) | 15시 → 15:00 |
| `null` + hour 1~12 | **받은 시각 이후 가장 가까운 쪽**(AM/PM). 둘 다 과거면 오후 기본 | 받은 7시 "3시"→15:00, "8시"→08:00 |

## 필드 가이드라인

### `title`
- 메시지에서 핵심 요약. 발신자 소속이 사실상 일정의 정체성이면 포함 (예: "서울내과 진료", "Korean Air KE093").

### `location`
- 장소가 명시된 경우만. 발신자 소속이 곧 장소면 발신자명을 location에 사용 가능 (예: 병원/식당 예약).

### `attendees`
- 메시지에 등장하는 **참석자**의 이름/별칭. 발신자(`<발신자:>`) 자신은 보통 포함하지 않음 (단, "팀장님이 호스트"처럼 발신자가 명시적 참석자로 언급되면 포함).

### `organizer`
- 발신 **기관/소속/주최**가 메시지에 명시된 경우만 채운다 (예: `서울내과`, `Lufthansa`, `구글코리아`). 없으면 `null`.
- 개인 발신(친구·동료 이름)은 `organizer`가 아니라 `attendees`/`title`로. organizer는 "조직"일 때만.

### `description`
- 부가 메모. 다음 정보들이 들어간다: **전화번호, 이메일, URL, 변경/취소 안내, 준비물, 드레스코드, 항공편 메모** 등.
- 메시지에서 직접 인용하거나 짧게 요약. 메시지에 없는 정보는 절대 추가 금지.

### `recurrence`
- RRULE 표기. 예: `FREQ=WEEKLY;BYDAY=TU` (매주 화요일), `FREQ=MONTHLY;BYMONTHDAY=1`.
- 일회성 일정이면 `null`.

### `confidence` (모델·gold 공통 스케일)
| 범위 | 케이스 |
|---|---|
| **0.90 ~ 1.0** | 날짜·시간·장소 모두 명시된 확정 일정 |
| **0.75 ~ 0.90** | 핵심 정보(날짜+시간) 있고 부가 정보 일부 누락 |
| **0.60 ~ 0.75** | 상대 시간만 있거나 장소 모호, 추론 일부 필요 |
| **0.40 ~ 0.60** | 부분 정보만 있고 추론 비중 큼 (시간만 또는 날짜만) |
| **`has_schedule: false`** | confidence 필드 자체 생략 또는 N/A — events가 빈 배열이므로 의미 없음 |

## Title 조합 규칙 (앱이 조합 — 모델은 활동만)

★ **모델은 freeform 제목을 쓰지 않는다.** `title`에는 **핵심 활동/주제만** 추출(예: `저녁식사`, `주간 회의`, `진료`, `Kickoff`).
"누구와"·"보낸사람(소속)"은 입력 메타데이터라 **앱 `compose_title`(= `_common.compose_title`)이 조합**한다.

조합 규칙:
1. **누구와**: `attendees`가 제목에 없으면 앞에 `"{참석자}와/과 {활동}"` (받침에 따라 와/과)
2. **출처(소속)**: 끝에 ` · {발신자}` 부착
   - 외부 개인 + 소속 있으면 → ` · {발신자} ({소속})`  (예: `· Sarah Lee (Company)`)
   - 기관 발신(발신자=소속) → ` · {기관}`  (예: `· 서울내과`)
   - 본인(`나`/`Me`) → 발신자 표시 생략

| 입력 | 모델 `title` | 최종 표시 제목 |
|---|---|---|
| 민지 카톡 "저녁 먹자" | 저녁식사 | `민지와 저녁식사` |
| 박과장 "주간회의" | 주간 회의 | `주간 회의 · 박과장` |
| Sarah Lee gmail(Company) | Kickoff | `Kickoff · Sarah Lee (Company)` |
| 서울내과 SMS | 진료 | `진료 · 서울내과` |
| 나 "등산" | 등산 | `등산` |

- **장소**: 제목에 넣지 않고 `location` 필드 → 캘린더 장소칸으로 표시(누락 금지).
- `organizer`(소속) 식별: 카톡/문자 발신자명 그대로 / Gmail은 도메인→회사명(`@company.com`→`Company`) / 본인은 `나`·본인이메일.

### 데이터 페어 구조 (학습 데이터 형식)

JSONL 한 줄에 다음 객체:

```json
{
  "scenario_id": "string",
  "received_at": "ISO8601",
  "channel": "kakao|sms|gmail",
  "sender": "string",
  "language": "ko|en|mixed_ko_en",
  "thread_context": [               // optional, 멀티턴 케이스에만
    {"time": "HH:MM", "sender": "string", "message": "string"},
    ...
  ],
  "message": "string",
  "gold": { "has_schedule": ..., "events": [...] }
}
```

`thread_context`가 없거나 빈 배열이면 단일 메시지 케이스. 있으면 멀티턴.

## 입력 포맷 (학습/추론 시)

### 기본 (단일 메시지)

```
<채널: KakaoTalk>
<수신시각: 2026-05-25T14:30:00+09:00 (월)>
<발신자: 김부장>
<메시지>
내일 3시에 회사 3층에서 주간회의 잡았습니다. 박과장도 같이.
</메시지>
```

- `<수신시각>`에는 요일 `(월)`만 붙인다. **상대 날짜를 모델이 계산하지 않으므로** round5의 `<날짜힌트>`(절대일자 미리 계산)는 폐기됨 — 모델은 "내일"을 그대로 `date:"내일"`로 추출하고, 절대일자 계산은 resolver가 한다.
  - 위 메시지의 기대 출력(일부): `"date": "내일", "time": {"hour": 3, "minute": 0, "marker": null}` → resolver가 2026-05-26 + (받은 14:30 기준 "3시"→15:00)로 변환.

### 확장 (멀티턴 — 대화내역 포함)

카톡/문자/Gmail의 일정 협의가 여러 메시지에 걸쳐 진행되는 경우, 안드로이드 앱이 직전 3~5개 메시지를 함께 입력:

```
<채널: KakaoTalk>
<수신시각: 2026-05-25T10:35:00+09:00 (월)>
<발신자: 나>
<대화내역>
[10:02] 김용안: 줌으로 한다면, 언제 가능하실까요?
[10:04] 나: 29일 오전, 30일 토 오전, 6/1 종일
[10:07] 나: 30일 토요일 오전에 줌으로 할까요?
[10:14] 김용안: 6월4일 오전 9시 어떤지요?
</대화내역>
<메시지>
좋습니다
</메시지>
```

### 입력 포맷 규칙

- `<대화내역>`은 **optional**. 단일 메시지로 의미가 명확하면 생략 (블록 자체를 넣지 않음).
- 대화내역 표기 형식: `[HH:MM] 발신자: 메시지` 한 줄씩. 시간순 (오래된 것부터).
  - ★ 이 형식은 `scripts/_common.py:build_user_block`이 학습·평가·(추후 앱) 공용으로 렌더링한다. 형식을 바꾸면 세 곳을 함께 바꿔야 함 (train/추론 분포 불일치 주의).
- `<수신시각>`에는 한국어 요일 `(월)`을 부착 (단일/멀티턴 동일).
- 보통 직전 3~5개. 너무 길면 모델 부담 (300토큰 권장).
- 본인 발신자는 `나` (ko/mixed) 또는 `Me` (en).
- 최종 메시지(가장 아래 `<메시지>`)가 모델이 일정 추출을 시도하는 **대상**. 대화내역은 그 메시지를 해석하기 위한 참고.

### 모델 판단 가이드 (멀티턴)

- 최종 메시지가 **확정 응답** (예: "좋습니다", "OK", "그렇게 하시죠", "네 알겠습니다")이면 스레드 맥락에서 직전 제안된 시각·장소를 추출해서 `has_schedule=true`.
- 최종 메시지가 **새 제안**이면 `has_schedule=false` (다음 확정 응답을 기다림).
- 스레드 맥락에 여러 시각 후보가 나왔다면 **가장 최근 제안된 것**이 최종 합의 시각.
- 최종 메시지가 **취소·변경 통보**면 스레드 맥락의 기존 합의와 비교해서 변경/취소 처리.

→ 모델은 위 스키마 JSON을 출력해야 함.
